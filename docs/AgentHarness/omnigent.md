# Omnigent

## Overview

Omnigent is an open-source **meta-harness** released by Databricks: a layer that sits above the agent harnesses teams already use — Claude Code, Codex, Cursor, Pi, and custom agents — and makes them interoperable parts of a single, governed system. Where a harness (see [Agent Harness](./agent-harness.md)) wraps a model to make it useful, a meta-harness wraps *harnesses*, giving teams one place to compose multiple agents, enforce policy across them, and collaborate on their sessions live. Omnigent is published at [omnigent-ai/omnigent](https://github.com/omnigent-ai/omnigent) under the Apache-2.0 license (~3.5k GitHub stars, 395 forks, mid-2026).

**Naming note:** Omnigent's "meta-harness" branding is distinct from the *Meta-Harness* automated harness-search method (Lee et al., arXiv:2603.28052) already documented in [Harness Optimization](./harness-optimization.md). The two share a name but not a mechanism — Lee et al.'s Meta-Harness is an algorithm that searches over harness code; Omnigent is a runtime product that orchestrates and governs existing, separately-built harnesses.

## Key Insight

However each agent harness calls into its LLM internally, the interface it exposes to users is the same: messages and files in, text streams and tool calls out. Omnigent builds a common API around this observation, wrapping both terminal-based coding agents (Claude Code, Codex, Pi) and SDK-based agents (OpenAI Agents SDK, Claude Agent SDK) behind one uniform session interface.

## Three Core Capabilities

| Capability | Description |
|---|---|
| **Composition** | Combine models, harnesses, and techniques without rewriting code; switch between Claude Code, Codex, Pi, and custom agents with one-line changes; delegate work to parallel sub-agents and run cross-vendor code review |
| **Control** | Stateful, contextual policies enforced at the meta-harness layer (not via prompts) — e.g., require human approval for a `git push` after an agent installs a new npm package, pause an agent after every $100 spent, or restrict an agent to writing only documents it created |
| **Collaboration** | Share a live agent session by URL so teammates can watch, chat, comment, and steer in real time; attach a co-driver for pair programming; fork a session into an independent branch |

## Architecture

| Component | Role |
|---|---|
| **Runner** | Wraps any agent in a sandboxed session behind a uniform API |
| **Server** (optional) | Centralizes policies and session sharing; exposes every session over terminal, native app, and web APIs; supports multi-user accounts and OIDC authentication (Google, GitHub, Okta, Microsoft) |
| **Harness adapters** | Pluggable execution runtimes per agent type (`claude-sdk`, `codex`, `claude-native`, etc.) — agents can be swapped without rewriting orchestration code |
| **Agent YAML spec** | Declarative agent definitions (prompts, tools, optional sub-agents), authorable by humans or generated by other agents |

### Sandboxing

Omnigent includes a flexible OS-level sandbox that can lock down OS access and intercept/transform outbound network requests — for example, never letting an agent see a GitHub access token directly, instead injecting it only at an egress proxy for pre-approved requests. Local sandboxing uses `bubblewrap` on Linux and the built-in `seatbelt` mechanism on macOS; cloud-hosted, disposable sandboxes are available via Modal, Daytona, and Islo, removing the need for local sandbox infrastructure. See [Agent Sandboxing](../SecurityFrameworks/agent-sandboxing.md) for the broader isolation taxonomy these mechanisms fit into.

### Policy Governance

Policies are contextual and stateful — they track dynamic state within a session (e.g., "has this agent installed a new dependency since the last approved push?") rather than relying on the model to self-police via prompt instructions. This places governance in the control plane rather than the context window, consistent with the authority-hierarchy principle in [Agent Harness](./agent-harness.md#authority-hierarchy).

## Installation

Requires Python 3.12+. Install via:

```
uv tool install omnigent
brew install omnigent-ai/tap/omnigent
```

Dependencies: `uv` (required), `git`, Node.js 22+ (for coding harnesses), `tmux`, and platform sandboxing tools (`bubblewrap` on Linux, built-in `seatbelt` on macOS).

## Best Practices

| Challenge / Area | Description | Solution / Recommendation |
|---|---|---|
| Vendor lock-in across coding agents | Teams standardize on one harness and lose access to other models' strengths | Use Omnigent's harness adapters to swap or run Claude Code, Codex, Cursor, and Pi side by side without rewriting orchestration logic |
| Prompt-based guardrails are unreliable | Relying on system-prompt instructions to enforce spending or approval limits is easily bypassed by adversarial inputs or model drift | Enforce contextual policies at the meta-harness layer (Runner/Server), not in the prompt |
| Secrets exposed to agent context | Tokens passed directly to agents risk leaking via prompt injection or logs | Use the egress-proxy sandbox pattern: inject credentials only at the network boundary for pre-approved requests |
| Reviewing agent work asynchronously only | Teammates only see agent output after the fact, missing chances to steer mid-task | Use live session sharing and co-driver attachment to review and steer in real time |
| Local infrastructure burden for sandboxing | Running isolated environments locally requires setup and maintenance | Use disposable cloud sandboxes (Modal, Daytona, Islo) instead of local sandbox infrastructure |
